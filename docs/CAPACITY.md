# Производительность, задержки и насыщение очереди

Документ отвечает на: **какая задержка у пользователя** и **когда начнутся
проблемы с очередью**. Все «software»-числа — замеры на текущем стенде (Docker
Desktop, Windows), «GPU»-числа — инженерные оценки (зависят от модели/карты).

---

## 1. Из чего складывается задержка пользователя

```
Пользователь → POST /generate → [очередь] → [GPU генерация] → callback/поллинг
                  ~12 мс         W (ожидание)     G (генерация)
```

Полное время до результата ≈ **W (ожидание в очереди) + G (генерация)**.
Накладные расходы API ничтожны по сравнению с G.

### Замеренный software-оверхед (на стенде)

| Операция | Медиана | p90 |
|---|---|---|
| `POST /generate` (валидация + запись в БД + enqueue) | **12.5 мс** | 14 мс |
| `GET /tasks/{id}` (поллинг статуса) | ~40 мс* | — |
| `GET /health` | 3.7 мс | — |

\* включает оверхед сети Docker Desktop на Windows; на Linux-проде заметно ниже.

**Вывод:** сам сервис — не узкое место. Время определяет **G** (GPU) и **W**
(сколько ждали в очереди при нехватке GPU).

### Время генерации G (оценка)

| Тип | RTX 4090 | A100/L40S | H100 |
|---|---|---|---|
| Фото (FLUX/SDXL + InstantID, 1024²) | 4–8 с | 3–5 с | 2–4 с |
| Видео (i2v, ~5 с клип) | 2–5 мин | 1–2 мин | 40–90 с |

Дальше берём ориентиры: **фото G≈6 с**, **видео G≈180 с** (на 4090).

## 2. Пропускная способность одного GPU-воркера

Воркер обрабатывает задачи по одной (`concurrency=1`, `prefetch=1`).
Пропускная способность μ = 1 / G:

| Тип | G | μ (задач/мин) | теоретич. макс/сутки | **безопасно (≈70%)** |
|---|---|---|---|---|
| Фото | 6 с | 10 | ~14 400 | **~10 000 / сутки** |
| Видео | 180 с | 0.33 | ~480 | **~330 / сутки** |

«Безопасно» = с запасом, чтобы очередь не разрасталась (см. §3).

---

## 3. Когда начинаются проблемы с очередью

Ключевая величина — **загрузка** ρ = λ / (c·μ), где
λ — входящий поток задач, c — число GPU-воркеров, μ — задач/сек на воркер.

Среднее время ожидания в очереди растёт как **ρ / (1 − ρ)** — это гипербола:

| ρ (загрузка) | Ожидание W (в единицах G) | Состояние |
|---|---|---|
| 0.5 | ~1 × G | спокойно |
| 0.7 | ~2.3 × G | рабочий потолок |
| 0.8 | ~4 × G | уже чувствуется |
| 0.9 | ~9 × G | плохо |
| ≥ 1.0 | **→ ∞ (очередь растёт без предела)** | отказ |

**Правило:** пока суммарный поток λ ниже ~70–80% от суммарной мощности c·μ —
всё хорошо. Как только λ приближается к c·μ, ожидание взрывается, а при λ > c·μ
очередь растёт **бесконечно** (задачи копятся быстрее, чем обрабатываются).

### Конкретно — где «потолок» на 1 GPU

- **Фото (μ=10/мин):** комфортно до ~7 задач/мин (~420/час). Всплески выше
  гасит очередь, но **устойчивый** поток >10/мин → бесконечный backlog.
- **Видео (μ=0.33/мин):** комфортно до ~0.23/мин (~14/час). Устойчиво >20/час →
  backlog.

### Всплески vs устойчивый поток
Очередь нужна, чтобы пережёвывать **всплески**: 100 задач разом на 1 GPU-фото
разойдутся за ~10 минут. Проблема не во всплеске, а в **среднем потоке выше
мощности** — его никакая очередь не спасёт, только добавление GPU.

## 4. Сколько GPU под нагрузку (ориентир)

Считаем по «безопасной» пропускной способности из §2 (поток усреднён; при
пиках ×2 закладывай вдвое больше или полагайся на автоскейл).

| Нагрузка | Фото-GPU | Видео-GPU |
|---|---|---|
| 1 000 / сутки | 1 | ~3 |
| 10 000 / сутки | 1–2 | ~30 |
| 50 000 / сутки | ~5 | ~150 |
| 100 000 / сутки | ~10 | ~300 |

Видно, **почему видео дорогое**: одна видеозадача = ~30 фото по времени GPU.
Поэтому видео вынесено в отдельную очередь (`generation_video`) и масштабируется
отдельно — чтобы не блокировать фото.

---

## 5. Что упрётся раньше GPU? (почти ничего)

| Компонент | Когда станет узким местом |
|---|---|
| **GPU** | первым — это главный и самый дорогой ресурс |
| API (FastAPI) | ~12 мс/запрос; одна реплика тянет тысячи enqueue/мин, масштабируется по CPU (HPA) |
| PostgreSQL | вставки задач — тысячи/сек; до десятков млн строк без проблем (потом — партиционирование/архив) |
| Redis (очередь/лимиты) | десятки тыс. оп/сек; не упрётся раньше GPU |
| Хранилище R2/S3 | практически безгранично; следи только за объёмом и ценой |

То есть масштаб упирается в **деньги на GPU**, а не в код.

---

## 6. Как система сама держит очередь под контролем

- **Раздельные очереди** `generation` (фото) и `generation_video` (видео).
- **KEDA** добавляет воркеры по длине очереди в Redis: порог `listLength`
  (фото 2, видео 1 на воркера) держит ρ в безопасной зоне, пока есть GPU.
- **HPA** масштабирует API по CPU.
- **Метрика `queue_size`** + алерт `QueueBacklog` (>100 в течение 10 мин) и
  `SlowGeneration` (p95 > 5 мин) предупредят заранее.
- **Таймауты + DLQ** не дают «застрявшим» задачам копиться.

### Практический порог тревоги
Следи за `queue_size` и p95 `generation_seconds`. Если `queue_size` устойчиво
растёт (не возвращается к ~0 между пиками) — λ превысил мощность, **добавляй
GPU-воркеры** (или поднимай лимит реплик KEDA, если упёрся в `maxReplicaCount`).

---

## 7. Как уменьшить задержку пользователю

- **Callback вместо поллинга** — результат прилетает сразу, без задержки опроса.
- Держать **ρ ≤ 0.7** (запас мощности) — тогда W ≈ G, ожидание почти нулевое.
- Для фото — батчинг в одном workflow (если модель позволяет) повышает μ.
- Видео — на более мощных картах (A100/H100) G падает в 2–3 раза.

> Все GPU-числа — оценки; уточни их одним прогоном реального workflow на твоей
> карте (время одной генерации G) и пересчитай §2–§4 под свои значения.
